## Authority, scope, and fixed boundaries

- [ ] 1. Bind implementation to `ROADMAP.md` Phase 7 and preserve its Phase 8 security gate.
  - [ ] 1.1 Keep `cmd/apparat` and `cmd/apparatd` as alternative compositions of one embedded, headless-capable core.
  - [ ] 1.2 Keep GUI navigation, focus, selection, layout, scrolling, gestures, animation, modal state, and unsaved widget text outside the core.
  - [ ] 1.3 Make SQLite authoritative for durable Phase 7 state behind repositories and the internal application API.
  - [ ] 1.4 Make the GUI call internal application commands and queries directly in-process without importing HTTP request/response types.
  - [ ] 1.5 Make REST handlers call the same internal application API without direct SQL access or REST-only product rules.

## Checkpoint A: shared node runtime and persistence ownership

- [ ] 2. Give both artifacts one logical default runtime root.
  - [ ] 2.1 Update `internal/config` so binary name no longer changes the default node root while explicit `--runtime-dir` and `APPARAT_RUNTIME_DIR` remain authoritative.
  - [ ] 2.2 Add a platform migration inspector that recognizes the legacy `apparat` and `apparatd` roots without merging two populated roots or overwriting either one.
  - [ ] 2.3 Migrate exactly one unambiguous legacy root to the new node root with atomic-rename-first behavior and actionable fallback guidance.
  - [ ] 2.4 Refuse automatic migration when both legacy roots contain state and report explicit recovery choices.
  - [ ] 2.5 Test new installs, explicit roots, one legacy GUI root, one legacy headless root, two conflicting roots, and interrupted migration recovery.
- [ ] 3. Implement exclusive one-node runtime ownership in `internal/platform`.
  - [ ] 3.1 Add a narrow runtime-lock interface with Unix/Android and Windows implementations plus build coverage for supported targets.
  - [ ] 3.2 Store safe lock metadata containing node root, PID, artifact, start time, and process mode without treating metadata alone as ownership.
  - [ ] 3.3 Acquire the lock before opening writable SQLite state or starting provider/API work.
  - [ ] 3.4 Return a typed contention error with owner metadata and non-destructive recovery guidance.
  - [ ] 3.5 Release the lock idempotently and test contention, stale metadata, double close, partial acquisition failure, and two independent explicit roots.
- [ ] 4. Move SQLite lifecycle into the adapter boundary and harden its operating contract.
  - [ ] 4.1 Replace `internal/database` ownership with documented SQLite connection, migration, transaction, diagnostic, integrity, and backup components under `internal/adapters/persistence`.
  - [ ] 4.2 Configure foreign keys per connection, one initial writable connection, a bounded busy timeout, cancellation-aware operations, and typed retry classification.
  - [ ] 4.3 Keep WAL disabled by default and make any future enablement an explicit evidenced option.
  - [ ] 4.4 Preserve checksumed forward migrations and improve duplicate version/name/order/checksum diagnostics.
  - [ ] 4.5 Add repository transaction helpers without exposing `*sql.DB` above the persistence adapter boundary.
  - [ ] 4.6 Add integrity-check and coordinated backup operations that record schema/app metadata and never copy an actively written database unsafely.
  - [ ] 4.7 Test backup creation, disposable-root restore validation, corrupt backup rejection, migration rollback, checksum mismatch, busy cancellation, and foreign-key enforcement.
  - [ ] 4.8 Update transitional cluster/messaging repositories to use the shared persistence boundary without broad Phase 8 behavior changes.
- [ ] 5. Make application startup and shutdown transactional.
  - [ ] 5.1 Split the current monolithic runtime file into documented lifecycle, composition, diagnostics, and public API files below the 400-line limit.
  - [ ] 5.2 Track explicit starting, ready, stopping, stopped, and failed lifecycle states.
  - [ ] 5.3 Start in order: paths/migration, lock, last-run and structured logging, SQLite inspection/migrations, identity validation, repositories, service manager, REST listener, background supervision, readiness.
  - [ ] 5.4 Unwind completed startup stages in reverse order after any failure and retain the primary error plus cleanup diagnostics.
  - [ ] 5.5 Stop admission/readiness first, then REST/background work, repositories, SQLite, logs, and the lock.
  - [ ] 5.6 Make concurrent and repeated close calls safe and deterministic.
  - [ ] 5.7 Test failure injection at every startup stage, ordered cleanup, readiness timing, signal cancellation, panic diagnostics, and repeated shutdown.

## Verification strategy

- Pure domain and application tests use fake clock, ID, repositories, drivers, and notifications.
- SQLite behavior uses disposable runtime roots and real `modernc.org/sqlite` integration tests.
- HTTP behavior uses `httptest` and ephemeral loopback listeners; tests never require remote networking.
- GUI state/projection tests remain display-free; native GUI and physical Android checks are separate evidence targets.
- Restart, contention, failure injection, backup/restore, and sanitization tests provide the primary reliability evidence.
- Full repository verification and build/smoke evidence remain the final gate.
